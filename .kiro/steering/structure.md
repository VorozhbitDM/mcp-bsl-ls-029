# Структура проекта

```
mcp-bsl-ls/
├── src/
│   └── mcp_bsl/               # Основной Python-пакет
│       ├── __init__.py        # Версия пакета
│       ├── server.py          # Точка входа MCP-сервера — регистрация инструментов, обработка запросов, форматирование ответов
│       ├── bsl_runner.py      # Обёртка подпроцесса BSL Language Server — запуск анализа/форматирования, парсинг вывода
│       └── config.py          # Загрузка конфигурации из переменных окружения, валидация путей (Pydantic-модель BSLConfig)
│
├── rules/                     # Инструкции для AI-агента (не Python-код)
│   ├── Project_rules.md       # Стандарты кодирования 1С/BSL для AI-агента
│   ├── User rules.md          # Общие инженерные принципы для AI-агента
│   └── mcp-tools.md           # Справочник по MCP-инструментам для AI-агента
│
├── .kiro/steering/            # Steering-файлы Kiro (эта директория)
├── .bsl-language-server.json  # Конфигурация BSL Language Server (язык, диагностики, форматтер)
├── mcp.json                   # Пример конфигурации MCP-клиента для Cursor/Kiro
├── pyproject.toml             # Метаданные проекта и зависимости
├── requirements.txt           # Зависимости pip
├── install.bat                # Скрипт установки для Windows (создаёт venv, устанавливает зависимости)
├── bsl-language-server-*.jar  # Исполняемый файл BSL Language Server (в идеале не коммитить в git)
└── venv/                      # Локальное виртуальное окружение (привязано к машине, в .gitignore)
```

## Ответственность модулей

| Модуль | Ответственность |
|---|---|
| `server.py` | Слой MCP-протокола — регистрирует инструменты `bsl_analyze` и `bsl_format`, диспетчеризует вызовы, формирует markdown-ответы |
| `bsl_runner.py` | Слой подпроцесса — строит команды `java -jar`, запускает их, парсит JSON/текстовый вывод в датаклассы `BSLDiagnostic` |
| `config.py` | Слой конфигурации — читает переменные окружения, валидирует пути к JAR и конфигу через Pydantic, предоставляет `get_config()` |

## Поток данных

```
MCP-клиент → server.py (вызов инструмента) → bsl_runner.py (подпроцесс) → BSL LS JAR → bsl-json.json → распарсенные диагностики → markdown-ответ
```

## Ключевые соглашения

- **Ленивая инициализация**: `BSLRunner` и `BSLConfig` создаются при первом вызове инструмента, а не при старте сервера
- **Файл vs директория**: оба инструмента принимают путь к файлу и молча перенаправляют на родительскую директорию для совместимости с BSL LS
- **Файл JSON-отчёта**: `bsl_analyze` читает `bsl-json.json`, записанный BSL LS в рабочую директорию; файл удаляется после чтения (сохраняется в режиме `DEBUG`)
- **Безопасное окружение**: подпроцесс запускается с `APPDATA`, `LOCALAPPDATA`, `TEMP`, `USERPROFILE`, `HOME`, перенаправленными в `tempfile.gettempdir()`, для обхода проблем с Windows junction point
- **Асинхронное выполнение**: анализ и форматирование запускаются в пуле потоков (`loop.run_in_executor`), чтобы не блокировать асинхронный event loop MCP
- **Логирование**: используется `logging` в `stderr`; уровень управляется переменной `BSL_LOG_LEVEL`; по умолчанию `WARNING`, чтобы не засорять stdio MCP
